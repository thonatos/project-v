## Context

当前 `generateHeadingId` 函数位于 `docs.ts`，使用正则 `/[^\w\s-]/g` 过滤字符。`\w` 仅匹配 `[a-zA-Z0-9_]`，中文字符被全部移除，导致中文标题生成空 id。

## Goals / Non-Goals

**Goals:**
- 支持中文标题生成有效 id
- 保持向后兼容（英文标题仍生成原有格式的 id）
- 无需引入外部依赖（如拼音转换库）

**Non-Goals:**
- 不实现拼音转换（复杂度高、依赖重）
- 不处理极端边界情况（如 emoji、特殊符号）

## Decisions

### 决策 1：保留中文字符而非拼音转换

**方案对比：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| A. 保留中文 | 无依赖、简单、可读性好 | URL 中显示中文（编码后） |
| B. 拼音转换 | 纯英文 URL | 需引入拼音库（~200KB），多音字歧义 |
| C. 数字序号 | 简单 | 不可读 |

**选择方案 A**：直接保留中文字符。现代浏览器完全支持 URL 中的 Unicode 字符（自动编码），无需拼音转换。

**实现方式：**
```typescript
function generateHeadingId(text: string): string {
  return text
    .trim()
    .replace(/\s+/g, '-')           // 空格转连字符
    .replace(/[^\p{L}\p{N}_-]/gu, '') // 仅保留字母、数字、连字符（支持 Unicode）
    .slice(0, 100);                  // 截断防止过长
}
```

`\p{L}` 匹配所有 Unicode 字母（包括中文），`\p{N}` 匹配所有数字。`u` flag 启用 Unicode 模式。

### 决策 2：唯一性通过计数器实现

若文档内有重复标题，第二个及之后的 id 添加 `-2`, `-3` 序号。在 `addHeadingIds` 中维护计数器 Map。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| URL 中文编码后较长 | 截断至 100 字符 |
| 旧文档英文 id 格式不变 | `\p{L}` 包含英文，向后兼容 |
| 空标题返回空 id | 保持当前行为，TOC 已处理空 id |
